# Agent 개발 Workflow — 초안

> 작성: 박동제 (pkdje) · 2026-05-30 · Week 1 산출물

이 문서는 "내가 AI 와 함께 어떻게 개발할 것인가"를 정리한다. 3주 안에서 점진적으로 발전시키고, 회고에서 무엇이 효과 있었는지 평가한다.

---

## 1. 핵심 원칙

1. **알고 통제하라.** AI 가 코드를 쓰지만 의도와 검증은 내가 한다. 합의 없이 진행되는 코드는 없다.
2. **단계를 분리하라.** 발산(브레인스토밍) → 수렴(설계) → 분해(태스크) → 구현 → 검증. 한 단계에 머무를 때 다른 단계로 도망가지 않는다.
3. **명시적 컨텍스트.** 매번 같은 정보를 다시 설명하지 않도록 MD 문서/Skill/메모리에 기록한다. 컨텍스트가 흐려지면 결과물 품질이 즉시 떨어진다.
4. **작게 자주 머지.** PR 하나에 1 ~ 2일 분량. 큰 PR 은 검토도 힘들고 AI 의 컨텍스트도 넘친다.
5. **시각적 결정은 시각적으로.** UI/UX 결정은 텍스트로 합의하지 않는다. 스크린샷이나 mockup 으로 본 다음 결정한다.

---

## 2. 개발 사이클 — 한 기능을 어떻게 만드는가

```
[1] 발산                     [2] 설계
브레인스토밍              ─►  데이터/알고리즘/UI
   (질문 1개씩)              섹션별 합의
                               │
                               ▼
[5] 검증                     [3] 분해
- 타입체크 OK               ─►  Issue/Task 로 쪼개기
- 기능 동작 직접 확인         (기능 단위 + 컴포넌트 단위)
- 회고 메모                    │
   ▲                          ▼
   │                       [4] 구현
   └────────────────────  feature 브랜치, 작은 PR
                          AI 가 초안, 내가 리뷰
```

### 단계별 산출물

| 단계 | 산출물 | 도구 |
|---|---|---|
| 1 발산 | 합의된 한 줄 설명 + 결정된 요구사항 | 대화 (질문 1개씩) |
| 2 설계 | 설계 섹션 (데이터 모델, 알고리즘, UI) | Markdown 문서, mockup |
| 3 분해 | GitHub Issue, 의존성 명시 | issue 템플릿 |
| 4 구현 | feature 브랜치 + PR | Claude Code 코드 작성 |
| 5 검증 | 동작 확인 + 회고 메모 | 브라우저, 테스트, 위키 회고 |

## 3. 작업 단위 쪼개기 기준

### 3.1 기능 단위 (vertical slice) — 우선
- 한 사용자 시나리오가 처음부터 끝까지 동작해야 1개의 기능 단위.
- 예: "사용자가 새 태스크를 만들 수 있다" → API + UI + 저장 + 화면 표시까지 한 묶음.
- PR 1개 = 기능 단위 1개. 1 ~ 2일 분량.

### 3.2 컴포넌트 단위 — 보조
- 기능 단위가 너무 클 때만 컴포넌트로 쪼갠다.
- 예: "DAG 시각화" 기능이 너무 크다 → (a) 노드 렌더 (b) 엣지 렌더 (c) 자동 레이아웃 으로 분할.
- 이 분할은 같은 기능 단위 안에서 PR 을 여러 개로 나누거나, 한 PR 안의 커밋 단위로 표현.

### 3.3 분해 체크리스트
한 task 를 issue 로 등록할 때:
- [ ] 한 줄 요약(제목)이 사용자/개발자 관점에서 동작을 설명하는가?
- [ ] Acceptance Criteria 가 명시되어 있는가? (3 ~ 5개)
- [ ] 다른 어떤 issue 에 의존하는가? (블로커 표시)
- [ ] 예상 PR 1개로 끝낼 수 있는가? (1 ~ 2일)
- [ ] 검증 방법이 명확한가? (어떻게 "됐다" 를 확인할지)

## 4. AI 요청 프롬프트 패턴

### 4.1 컨텍스트 주입 (매 요청에 포함)
1. **무엇을 하려는가** — 한 문장. 예: "Task CRUD API 구현".
2. **왜** — 이게 어떤 시나리오를 풀려는 건지. 예: "사용자가 새 태스크를 만들고 편집/삭제할 수 있어야 한다."
3. **이미 결정된 것** — 데이터 모델, 스택. 매번 다시 설명하지 않고 문서 참조 (예: `docs/planning/01-project-plan.md` 의 4장).
4. **이번에 결정해야 할 것** — 명시. 예: "라우트 구조와 입력 검증 방식".
5. **하지 말 것** — 명시. 예: "Mongoose 쓰지 말 것, 인증 추가하지 말 것".

### 4.2 요청 유형별 템플릿

**(A) 발산용 — 브레인스토밍**
> "X 를 하고 싶다. 접근 방식 2 ~ 3가지를 trade-off 와 함께 제안해줘. 추천도 같이."

**(B) 수렴용 — 설계 의사결정**
> "지금까지 결정: [A, B, C]. 이번에 결정할 것: D. 옵션 [(i), (ii), (iii)] 중 어느 쪽? 한 번에 한 질문씩."

**(C) 분해용 — Task 쪼개기**
> "이 기능을 구현하기 위한 PR 단위 task 목록을 만들어줘. 각 task 의 acceptance criteria 와 의존성 포함."

**(D) 구현용 — 코드 작성**
> "[문서 경로] 에 정의된 X 를 구현해줘. 이미 작성된 [관련 파일 경로] 참고. 새 의존성 추가 시 먼저 알려줘."

**(E) 검증용 — 코드 리뷰**
> "이 PR 의 변경사항을 리뷰해줘. 1) 요구사항 누락 2) 잠재 버그 3) 단순화 가능 영역 순으로."

### 4.3 안티패턴
- ❌ 한 번에 여러 결정 요구 ("API 도 짜고 UI 도 짜고 추천도 만들어줘")
- ❌ 컨텍스트 없이 코드만 요청 ("Task CRUD API 짜줘")
- ❌ "알아서 해줘" — 의도를 위임하지 않음
- ❌ AI 의 첫 번째 출력을 무비판 수용 — 항상 한 번 더 검토

## 5. 내가 직접 검증해야 할 체크포인트

### 5.1 매 PR 마지 전
- [ ] **요구사항 매칭** — issue 의 acceptance criteria 와 1:1 대응되는가
- [ ] **타입체크 / lint** 통과
- [ ] **수동 동작 확인** — 브라우저에서 실제로 클릭/입력해서 시나리오를 처음부터 끝까지 돌려봄
- [ ] **불필요한 코드** — AI 가 "혹시 모르니" 추가한 에러 핸들링/추상화는 제거
- [ ] **데이터 모델 변경 영향** — 다른 기능을 깨지 않는가 (Week 2 부터는 특히 주의)
- [ ] **커밋 메시지** — 한 줄 요약 + 본문 (왜 변경했는지)

### 5.2 매주 마지막
- [ ] **회고 작성** (위키) — 이번 주 무엇이 잘됐고 안됐는지, AI 활용 중 어떤 패턴이 효과적이었는지
- [ ] **워크플로우 문서 업데이트** — 이번 주 발견한 안티패턴/개선점 본 문서에 추가
- [ ] **백로그 재정렬** — 다음 주 우선순위 점검

### 5.3 의사결정 체크포인트 (AI 에게 위임하지 않을 결정)
- **스코프 변경** — 새 기능 추가, 기존 기능 제거
- **외부 의존성 추가** — 새 라이브러리 도입
- **데이터 모델 변경** — 필드 추가/삭제/타입 변경
- **사용자 경험 결정** — 인터랙션 방식, 시각적 디자인 방향

## 6. 재사용 가능한 Skill / MD 파일

### 6.1 본 프로젝트에서 만들거나 활용할 Skill

| Skill / 문서 | 용도 | 상태 |
|---|---|---|
| `superpowers:brainstorming` | 발산 단계, 단계별 설계 합의 | 활용 중 |
| `superpowers:writing-plans` | 설계 → 구현 계획 변환 | 활용 예정 |
| `superpowers:test-driven-development` | 추천 알고리즘 같이 로직 명확한 부분에 적용 | Week 2 |
| `superpowers:verification-before-completion` | PR 마지 전 동작 확인 강제 | Week 2 |
| `docs/planning/*.md` | 결정 누적, 매 요청의 컨텍스트 | 누적 중 |
| `.claude/memory/*` | 사용자 선호 (예: "auto-commit 금지") | 활용 중 |

### 6.2 새로 만들 만한 Skill (필요해지면)
- **"flowtodo-task-issue"** — issue 본문 템플릿(요약, AC, 의존성, 검증 방법) 을 일관되게 생성.
- **"flowtodo-pr-review"** — PR 리뷰 시 5.1 체크리스트 자동 적용.
- **"flowtodo-retrospective"** — 회고 작성 시 항상 같은 4개 질문(잘된 점, 안된 점, 배운 점, 다음 주 할 것) 으로 가이드.

이건 Week 2 이후 동일한 작업을 3번 이상 반복하게 됐을 때 만든다 — YAGNI.

## 7. 회고에서 평가할 것

- 한 사이클(발산→구현→검증) 평균 시간이 얼마인가?
- 어떤 프롬프트 패턴이 가장 효과 있었는가?
- AI 가 잘못된 방향으로 간 사례 — 무엇이 신호였는가?
- 5.3 의 의사결정 체크포인트를 실제로 지켰는가?
- 다음 주에 이 워크플로우 어디를 바꿀 것인가?
